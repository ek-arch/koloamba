## Open design calls (not yet decided / not yet built)

### 1. Telegram auth (alongside X OAuth)
NextAuth v5 has no first-party Telegram provider because Telegram uses the Login Widget with an HMAC-signed callback (not OAuth2). Cleanest path: a custom `CredentialsProvider` that verifies the Telegram hash server-side on the Next.js server.

Schema implications:
- `users.twitter_handle` becomes **nullable**
- add `users.telegram_id` (also nullable)
- either identifier can be the sign-in key; optional link-accounts flow later
- RLS and the admin `requireStaffPage()` guard continue to work off `users.id` / `role` — unaffected

### 2. Point balance from the Kolo replica
User intends to expose a read-only Postgres replica of the existing Kolo product so this app can show each ambassador's live point balance. **Decision pending: live-query vs periodic sync.**

- **Live-query**: always fresh; adds a hop + replica dependency on every render. Mitigate with a 60s SWR cache keyed by identifier.
- **Sync into `users.old_points`**: fast reads, decoupled from replica uptime, but stale up to sync interval.

Identifier to key on depends on how the replica represents users (telegram_id? twitter_handle? internal id?) — waiting on replica schema from user.

Balance powers: leaderboard display, tier computation (currently driven by `old_points` thresholds).

### 3. Bulk CSV import for `old_points`
Proposed flow (not built): admin uploads `handle,old_points` CSV → upsert into a new `pending_old_points` staging table keyed by handle → for handles already in `users`, apply + recompute tier immediately; for not-yet-signed-in handles, sign-in flow reads staging and applies on first login.

May become moot if the replica path (design call 2) is adopted, since the replica would be the source of truth.
